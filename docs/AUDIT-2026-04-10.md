# Darbitex V1 — Full Audit, 2026-04-10

All line numbers reference V1 source at `contracts/sources/` as of commit
`7c729e1`. The package is live on Aptos mainnet at publisher
`0x810693eb5e17185ee7d80e548a48edcb60be4b1d56d33f8c1be716d9fb422d2e`,
threshold 3-of-5, with three pools holding about $4 of liquidity. The audit
was run immediately after smoke tests passed on all three pools.

## Severity summary

| Severity | Count | Blocker for TVL growth? |
|---|---|---|
| CRITICAL | 0 | — |
| HIGH | 4 | yes — upgrade before TVL > ~$1k or external LPs |
| MEDIUM | 6 | should-fix, not emergency |
| LOW | 7 | nice-to-have |
| INFO | 5 | documentation only |

No fund-theft vectors were found. The HIGH issues are a denial-of-service
on small swaps, a pair of HookCap lifecycle desyncs, and a cosmetic const
deviation. None of them allow a current upgrade-key or admin-key holder
to steal LP funds today.

## HIGH findings

### H1. Swap underflow DoS for `amount_in < 10_000` raw

**Location:** `pool.move:412-432`

The fee computation has two independent floors that disagree at the boundary:

```move
let extra_fee = amount_in / EXTRA_FEE_DENOM;                // = 0 for amount < 100k
let extra_fee = if (extra_fee == 0 && amount_in > 0) { 1 } else { extra_fee };
// ... on a plain pool:
protocol_fee = extra_fee;                                   // = 1
let total_fee = amount_in * SWAP_FEE_BPS / BPS_DENOM;       // = 0 for amount < 10k
let lp_fee = total_fee - hook_fee - protocol_fee;           // 0 - 0 - 1 = u64 underflow
```

For any `amount_in` in `[1, 9_999]` the subtraction on the last line
underflows and the transaction aborts. The intent of the `extra_fee` floor
was to "prevent zero-fee dust swaps"; in practice it *causes* dust swaps to
abort instead.

The boundary was verified against on-chain data from the smoke tests:
- `amount_in = 10_000` USDT in the USDT/USDC pool → `protocol_fee = 1`
  raw; swap succeeded (`total_fee = 1`, `lp_fee = 0`, no underflow).
- `amount_in = 1_000_000` octas APT in both APT pools → `protocol_fee = 10`
  octas; swap succeeded (`total_fee = 100`, `lp_fee = 90`).
- `amount_in = 9_999` would underflow `lp_fee` to u64 max and abort.

**Impact.** All swaps with `amount_in` below 10 000 raw units revert. For a
6-decimal stable that is below $0.01; for APT (8 decimals) it is below
0.0001 APT. No funds at risk, but it is a silent DoS that will surface as
opaque aborts in any frontend, arbitrage bot, or hook that routes small
amounts through the pool.

**Fix (minimal, compatible upgrade):**

```move
let lp_fee = if (total_fee > hook_fee + protocol_fee) {
    total_fee - hook_fee - protocol_fee
} else { 0 };
```

`lp_fee` is only consumed by the `Swapped` event; the zero substitution is
cosmetic. The underflow is what needs to go.

### H2. HookCap resurrection after `remove_hook` + same-address `set_hook`

**Location:** `pool.move:196-199`, `261-284`, `288-297`

`remove_hook` clears `hook_addr` and resets `hook_claimed = false`, but it
does not invalidate previously minted `HookCap` values. `HookCap` has the
`store` ability (not `drop`), so a hook module can persist it in its own
storage indefinitely.

If governance later calls `set_hook` with the *same* address — during a
resale back to the original hook owner, after `admin_force_remove_hook`, or
during a second auction won by the same party — `assert_valid_cap` passes
again because it only compares `cap.hook_addr == pool.hook_addr`. There is
no per-auction epoch, so the previously-ejected hook silently regains
`swap_hooked`, `flash_borrow`, and `withdraw_hook_fee` rights.

**Impact.** A hook that was previously removed can resume full privileges
on re-assignment. With current V1 state (three plain pools, zero hooked)
the bug is inert — no `HookCap` has ever been minted — but it is a governance
foot-gun the moment the first hook is auctioned.

**Fix sketch.** Add a `hook_epoch: u64` field to `Pool` and `HookCap`, bump
it in `set_hook` and `remove_hook`, stamp it at `claim_hook_cap`, check it
in `assert_valid_cap`. This is **not** a compatible upgrade — `Pool` and
`HookCap` both get a new field — so it requires either a breaking redeploy
or an alternative that stores the epoch in a separate new resource keyed
by `pool_addr`. The alternative still cannot invalidate extant `HookCap`
instances without a new assertion path. Current live state has zero caps,
so a breaking fix is still costless if applied before the first hook
auction.

### H3. `hook_wrapper` lifecycle desync on consecutive @darbitex auctions

**Location:** `hook_wrapper.move:48-63` (`register_pool`), `137-147`
(`unregister_pool`)

This is the same family as H2, observed from the wrapper side. The
scenario:

1. Auction #1 awards the hook to `@darbitex`; the wrapper calls
   `register_pool`, which stores a `HookCap` in `WrapperRegistry.caps[pool]`.
2. The hook is later removed — via resale, `admin_force_remove_hook`, or
   a future path — clearing `pool.hook_addr`.
3. Auction #2 is held and won by `@darbitex` again; `set_hook` re-assigns
   the pool's hook.
4. The new winner tries `register_pool`. `table::contains(caps, pool_addr)`
   is still true from step 1, so the call aborts with
   `E_ALREADY_REGISTERED`.
5. `unregister_pool`'s staleness check (`hook_opt != @darbitex`) cannot
   fire because the pool's hook is `@darbitex` again — so the wrapper
   cannot unregister the stale entry either.

The pool is stuck serving swaps through a cap minted under an earlier
auction until the hook address is swapped to a different package.

**Impact.** The second auction winner is economically bricked: they paid
the bid but cannot execute hook logic via the wrapper. No theft.

**Fix.** H2's epoch fix resolves this too: the wrapper's stored cap
carries an old epoch, `unregister_pool` accepts an epoch mismatch as a
staleness signal, and the new winner can register.

### H4. `POOL_SEED_PREFIX` still reads `b"darbitex_pool_v3"` — accepted deviation

**Location:** `pool_factory.move:23`

The V1 clean-code spec asked for the seed to be renamed to
`b"darbitex_pool"`. It was not renamed. The three live mainnet pools have
their resource-account addresses derived from the v3 seed, so renaming now
would invalidate `canonical_pool_address()` for every existing pool.
**Accepted as-is** — the label is historical and carries no semantic
meaning in V1. A future breaking redeploy can rename it cleanly.

## MEDIUM findings

### M1. `claim_hook_cap<W>` witness auth by module address only

**Location:** `pool.move:288-297`

`witness_addr = type_info::account_address(&type_info::type_of<W>())`
authenticates the package, not a specific module. Any module at the
winning hook's package address can construct the witness type that passes
the check. Not exploitable today because well-written hook packages keep
their witness types module-private, but tightening is worthwhile.

**Fix.** Store `(address, module_name)` on `HookCap` and compare both at
claim and validate. Also requires struct layout change → breaking upgrade.

### M2. Bridge 80% imbalance cap too loose for pegged assets

**Location:** `bridge.move:166-170`

`MAX_IMBALANCE_PCT = 80` permits reserves to skew 80/20 before rejection.
For stablecoins advertised as pegged, a 20% price impact is catastrophic.
LPs can pre-skew via proportional `add_bridge_liquidity` (5% per add) and
walk reserves into extreme territory before a first swap arrives.

**Fix.** Lower the const to `60` (40/60 split) in the next compatible
upgrade. A proper Curve-style A-parameter stableswap curve is a larger
redesign that can wait.

### M3. `create_bridge` accepts arbitrary first-depositor ratio

**Location:** `bridge.move:86-130`

No check that `amount_from ≈ amount_to` at creation. An attacker can
front-run legitimate bridge creation by seeding `1_000_000:1`, then sweep
the expensive side at a near-zero price.

**Fix.** Require symmetric initial deposit within ~1% tolerance (a 101:100
split is fine, 1_000_000:1 is rejected). Compatible upgrade — only adds an
assert.

### M4. Router multi-hop missing sanity checks

**Location:** `router.move:94-137`

No assertion that `pool1 != pool2 != pool3`, no metadata-chain validation
that hop 1's output token matches hop 2's input token, no cycle detection.
A misrouted path fails deep in the call tree with an opaque error; a
crafted cyclic path could arbitrage asymmetric fees between the plain and
wrapped legs.

**Fix.** Add pool-distinctness asserts and a light metadata-chain check
per hop. Compatible upgrade.

### M5. `finalize_auction` no-bid resale lockout

**Location:** `pool_factory.move:293-298`, `start_auction`/`cancel_auction`

A seller who lists a hook for resale cannot unwind their listing before
`end_time` if no bids arrive — up to 30 days of lockout. Not fund-loss,
but a griefing UX wart.

**Fix.** Allow the seller to cancel their own resale listing before
`end_time` when `!has_bid`. Compatible upgrade.

### M6. Test coverage is 35% (22 of 34 public entries untested)

**Finding type:** quantitative gap, not a specific bug.

22 public entry functions have zero unit-test coverage in V1:

- Entire auction lifecycle: `start_auction`, `bid`, `cancel_auction`,
  `finalize_auction`
- All admin-gated entries: `pause`, `unpause`, `withdraw_protocol_fee`,
  `admin_force_remove_hook`
- Entire bridge surface: `add_bridge_liquidity`, `remove_bridge_liquidity`,
  `bridge_entry`, `pause_bridge`, `unpause_bridge`, `withdraw_fees`
- Entire hook wrapper: `register_pool`, `unregister_pool`,
  `hook_wrapper::swap_entry`, `add_liquidity`, `remove_liquidity`,
  `withdraw_fees`
- Router multi-hop: `swap_2hop`, `swap_2hop_mixed`, `swap_3hop`,
  `swap_3hop_mixed`, deadline variants

Several admin tests were deleted during the V1 refactor because hardcoded
ADMIN means the test harness cannot impersonate the multisig signer. The
fix is a `#[test_only]` admin override that tests can use to bypass
`assert_admin`, plus new tests for auction, bridge, wrapper, and router
paths.

**Recommendation.** Do not route external liquidity through this deploy
until test coverage reaches parity with alpha (which had all of these
paths covered before the refactor). Bundle the test work with the HIGH/M
upgrade patch.

## LOW findings

| # | Location | Description |
|---|---|---|
| L1 | `pool.move:308-351` | `withdraw_hook_fee` and `withdraw_protocol_fee` are not gated by `pool.locked`. The reserve accounting is currently consistent only by arithmetic coincidence with the flash-loan re-sync path. Add `assert!(!pool.locked, E_LOCKED)` at the top of both to remove the fragile coupling. |
| L2 | `pool.move:435-447` | `update_twap(pool)` is called *after* the reserve mutation in `swap_internal`. It should run *before*, matching the ordering used in `add_liquidity_internal` and `remove_liquidity_internal`. The current order biases the TWAP cumulative by the swap's own impact over the prior interval — a minor oracle integrity issue that becomes significant once a TWAP-consuming hook is deployed. |
| L3 | `bridge.move:139-190` | Reentrancy guard is half-implemented — `locked` is set and cleared in swap paths but absent from add/remove liquidity. Move's single-threaded VM makes this decorative in practice, but the asymmetry is confusing for future maintainers. |
| L4 | `bridge.move:256-257` | `remove_bridge_liquidity` can dust-lock the last LP if rounding puts `lp_supply` exactly at `MINIMUM_LIQUIDITY`. Minor UX issue, no loss. |
| L5 | `router.move:18-20` | Deadline check uses `<=` instead of `<`, allowing exact-second replay. Negligible. |
| L6 | `pool.move:600-660` | "Plain pools have no flash loans" is enforced only by the type system — the only constructor of `FlashReceipt` is `flash_borrow_hooked`, which requires a `HookCap`. A future addition of an alternate constructor would silently enable flash loans on plain pools. Worth a comment at the struct definition. |
| L7 | `pool_factory.move:164-165` | `create_canonical_pool` relies on `object::create_named_object` to abort on collision instead of explicit registry check. Not exploitable today. |

## INFO findings

- **Refactor is regression-free.** The diff review verified that every
  audit-tag comment stripped from alpha had its underlying assert, state
  update, or event emission preserved in V1. Twelve load-bearing sites
  were spot-checked: strict-k comparison, flash repay fee enforcement,
  add/remove liquidity `!locked` guards, `MINIMUM_LIQUIDITY` floor on
  burn, imbalance guard in bridge, proportional LP mint, `has_bid`
  finalize guard, anti-snipe extension, floor-price resale logic. All
  intact.
- **ADMIN and TREASURY consts match live multisigs character-for-character.**
- **`protocol_config()` tuple shape is preserved** — all callers
  (`bridge`, `hook_wrapper`, `pool_factory`) continue to destructure
  `(admin, treasury, factory_addr)` correctly.
- **`lp_coin` is clean** — friend-only mint/burn, soulbound (no transfer
  path), existence-checked table ops, safe `balance` view on missing
  registries.
- **First-depositor MINIMUM_LIQUIDITY lock** is implemented correctly and
  prevents the classic Uniswap V2 LP-inflation attack.

## Verdict

The package is **safe for current TVL** (~$4 across three smoke-test
pools) but **the HIGH findings and the test-coverage gap block meaningful
TVL growth**. Specifically:

- Do **not** open V1 to external LPs before shipping an upgrade that
  fixes H1 and adds coverage for the untested paths.
- Do **not** auction the first hook before a design is in place for H2
  and H3. A breaking redeploy that adds `hook_epoch` to `Pool` and
  `HookCap` is the cleanest path; an alternative lazy epoch resource is
  possible but clumsier.
- The HIGH findings are not emergency. No pause is required. There is no
  theft vector in the current state.

The `compatible` upgrade bundle drafted alongside this report fixes
**H1, L1, L2, M2, M3, M4, M5** in approximately 100 lines of diff across
three files and can be deployed in a single `3-of-5` multisig upgrade.
**H2, H3, M1, M6** are deferred — H2/H3/M1 because they require struct
layout changes and therefore a breaking redeploy, M6 because it is
ongoing test engineering work.
